1. Project Identity & Absolute Rules
1.1 What This System Is
This is a production-grade, multi-tenant SaaS Restaurant Management System. Every architectural decision, every line of code, every UI component must be built with production quality: fault-tolerant, secure, performant, maintainable, and scalable from day one. There is no MVP mode. There is no prototype mode. All features described in this document are in scope for the final shipped product.
1.2 Non-Negotiable Rules
✦ RULE: No tip management anywhere in the system. No tip field, no tip calculation, no tip UI, no tip database column. Completely absent.
✦ RULE: Staff accounts are created exclusively by the restaurant Admin. Staff cannot self-register, cannot sign up, cannot create their own accounts under any circumstance.
✦ RULE: Every tenant (restaurant) is fully data-isolated. No cross-tenant data access is ever possible.
✦ RULE: The Super Admin (platform owner) is the only entity that can create Admin-level restaurant accounts.
✦ RULE: The entire application UI, all labels, error messages, notifications, receipts, and PDFs must be available in French and Arabic with complete RTL layout support for Arabic.
✦ RULE: Node.js is the backend runtime. Fastify is the HTTP framework. No exceptions.
✦ RULE: All API responses on POS-critical paths must be under 200ms. All WebSocket events must propagate in under 500ms.
✦ RULE: The system must pass production security standards: HTTPS-only, JWT with rotation, RBAC on every endpoint, full audit logging, SQL injection prevention, input sanitization everywhere.
2. System Architecture
2.1 Overall Architecture Pattern
The system follows a three-layer architecture: a Flutter-based frontend (POS terminal app, back-office web app, mobile manager app), a Node.js/Fastify REST + WebSocket API layer, and a PostgreSQL/Redis/S3 data layer. All layers are containerized and horizontally scalable.
Frontend → Backend: REST API (authenticated via JWT Bearer tokens) for all CRUD operations
Frontend → Backend: WebSocket (Socket.io) for all real-time events: order updates, KDS tickets, table status, stock alerts
Backend → Database: PostgreSQL via `pg` connection pool (PgBouncer in front for connection management)
Backend → Cache: Redis for sessions, menu snapshots, rate-limit counters, real-time pub/sub relay, BullMQ job queues
Backend → Storage: S3-compatible object store (Cloudflare R2) for item photos, receipts, supplier documents, staff files
Backend → External: Payment gateway, SMS provider, email provider, aggregator webhooks
2.2 Multi-Tenancy Architecture
Each restaurant is an independent tenant identified by a UUID tenant_id. Tenant isolation is enforced at three levels:
Database level: every table that contains tenant data has a tenant_id column, indexed, and all queries include WHERE tenant_id = $1. PostgreSQL Row Level Security (RLS) policies are enabled as a secondary defense.
API level: every authenticated request extracts tenant_id from the JWT claims. Middleware validates that the resource being accessed belongs to the requesting tenant before executing any database query.
Storage level: all files are stored under a tenant-scoped S3 prefix: /tenants/{tenant_id}/...
⚠ A bug that allows cross-tenant data read is treated as a critical security incident, not a bug.

4. Database Schema — Full Production Design
4.1 Schema Principles
Every tenant-scoped table has tenant_id UUID NOT NULL with FK to tenants.id and a composite index on (tenant_id, id)
All primary keys are UUIDs (gen_random_uuid()) — no sequential integers exposed externally
All tables have created_at TIMESTAMPTZ NOT NULL DEFAULT NOW() and updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
Soft deletes via deleted_at TIMESTAMPTZ NULL — queries always include WHERE deleted_at IS NULL
Text fields for user-facing content have _fr and _ar variants: e.g., name_fr TEXT, name_ar TEXT
Money values stored as INTEGER (smallest currency unit, e.g., centimes) — never FLOAT or DECIMAL for currency
All migrations versioned and reversible via Knex migration files

5. API Specification
5.1 API Design Standards
Base URL: https://api.{tenant-slug}.rms.com/v1 OR https://api.rms.com/v1 with X-Tenant-ID header
All responses follow: { success: boolean, data: T | null, error: { code, message_fr, message_ar } | null, meta: { page, limit, total } | null }
HTTP status codes used correctly: 200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 429 Too Many Requests, 500 Internal Server Error
All endpoints protected by verifyJWT middleware except /auth/login, /auth/refresh, /reservations/public/:tenant
All endpoints enforce tenant isolation via verifyTenantAccess middleware
All write endpoints validate RBAC permissions via requirePermission('entity.action') middleware
Pagination: cursor-based for lists (after, limit), offset-based for reports
All list endpoints support filter, sort, and search query parameters

6. WebSocket Event Protocol
6.1 Connection & Rooms
Client connects with auth token: socket.auth = { token: accessToken }
Server middleware validates JWT on connection. Disconnects on invalid token.
On successful connection, server joins socket to rooms: tenant:{tenant_id}, branch:{branch_id}, station:{prep_station} (kitchen clients), table:{table_id} (waiter clients)
Each Socket.io namespace: /pos (FOH terminals), /kitchen (KDS screens), /manager (back-office and mobile app)

7. Frontend Architecture — Complete
7.1 App Structure
The Flutter codebase targets three surfaces: POS Terminal App (Android tablet/dedicated POS hardware), Back-Office Web App (browser, responsive), Mobile Manager App (iOS + Android phone)
Single monorepo with shared packages: core (models, API client, constants), ui (design system components), features (feature-by-feature modules)
Feature-first folder structure: lib/features/{feature_name}/{data,domain,presentation}
7.2 Design System
All UI components built from scratch in a shared ui package. No dependency on any third-party UI kit.
Design tokens: color palette, typography scale, spacing grid (4px base), elevation, border radius — all defined as Dart constants
Components include: RmsButton (primary, secondary, danger, ghost), RmsInput (with validation states), RmsCard, RmsModal, RmsBottomSheet, RmsTable, RmsStatusBadge, RmsCurrencyDisplay, RmsSearchBar, RmsDatePicker (RTL-aware), RmsTimePicker, RmsFloorPlanCanvas (custom painter), RmsKdsTicket, RmsReceiptPreview
All interactive components have loading, disabled, error, and success states
Tap targets minimum 48x48px per accessibility guidelines
⚠ No component should hardcode LTR layout. Always use start/end alignment, Directionality, and TextDirection.
7.3 Localization Architecture
Two ARB files: app_fr.arb and app_ar.arb — one key per UI string
All database-sourced user-facing strings (item names, category names, error messages from server) use the _fr or _ar field based on active locale
Currency display: uses NumberFormat.currency(locale: activeLocale, symbol: tenant.currency_symbol)
Date/time: uses DateFormat(locale: activeLocale). French: dd/MM/yyyy HH:mm. Arabic: uses Arabic calendar with Arabic-Indic digits (configurable per tenant).
RTL layout: Directionality widget at root. All Row → start/end. All padding/margin uses EdgeInsetsDirectional. All icons that imply direction (arrows, menus) flip with Directionality.
Locale preference stored per user in backend. Falls back to tenant default locale. Overridable per POS device (stored in Hive).
Receipts and PDFs: generated in the locale of the order's customer profile. Falls back to tenant default.

7.6 Offline Mode
POS terminal monitors connectivity via connectivity_plus and backend heartbeat ping every 30 seconds.
On connectivity loss: UI shows offline banner. POS continues operating using Hive-cached menu snapshot.
Orders created offline stored in Hive with status: pending_sync.
Payments in offline mode: cash only. Card payments are blocked with a visible message.
On reconnect: sync queue processes pending_sync orders in order. Conflict resolution: server is authoritative for table status.
Menu updates received while offline queued and applied on reconnect.
Offline mode event displayed in audit log with is_offline: true flag.
8. Feature Modules — Production Specification
8.1 Point of Sale & Order Management
Table Management
Floor plan is stored as JSON in the database (tables with x, y, width, height, shape).
Admin can edit the floor plan via a drag-and-drop canvas editor in back-office. Changes saved immediately.
Real-time table status pushed via WebSocket to all connected POS terminals in the same branch.
Table occupancy timer: starts when status transitions to occupied. Alert threshold configurable per tenant (default: 90 minutes). Timer displayed on floor plan card.
Merge tables: creates a logical merge record linking multiple table IDs to one primary order. All tables show as occupied. Unmerge splits back to individual bills.
Table transfer: reassign an open order from one table to another without closing.
Order Flow
Order creation validates: table is available (or delivery/takeaway), at least one item added, waiter has permission.
Every order state transition is recorded with timestamp, user_id, and previous state in audit_log.
Special requests and allergen alerts: any item flagged as containing an allergen category from the customer's dietary profile triggers an inline warning that the waiter must acknowledge before adding.
Course management: items assigned to courses 1-n. Waiter fires a course from POS → order_items for that course transition to fired status → WebSocket event to kitchen station.
Payment Processing
Payment session created when waiter taps Pay. Session tracks: total due, amount tendered so far, remaining balance.
Multi-tender: waiter can accept multiple payment methods against one order. Session closes when remaining balance reaches zero.
Cash: system calculates and displays change due. Drawer open command sent to cash drawer via ESC/POS.
Card payments: via payment gateway SDK. Terminal shows card prompt. On gateway callback: payment record created, order closed. Handles network timeout with idempotent retry.
Refunds: partial or full. Requires manager authorization code. Creates negative payment record. If card: gateway refund initiated. Updates order status.
Voids: only on orders not yet paid. Requires manager authorization. Restores table to available. Notifies kitchen to discard tickets.
No tip fields, tip prompts, tip calculations, or tip-related logic exist anywhere in the payment flow.
Discounts
Discount rules stored in database: name, type (percentage/fixed/bogo/category_pct/happy_hour), value, conditions_json (time range, minimum order, applicable categories), requires_manager_approval (bool), max_uses, expiry.
Happy hour rules: system checks current time against rule time range on every order. Auto-applies if conditions met.
Manual discount: cashier selects discount type and value. If requires_manager_approval is true, PIN prompt for manager. After authorization, discount logged with manager_id.
Coupon codes: validated against discount_codes table. Single-use codes marked redeemed on use. Redemption logged with order_id.
8.2 Kitchen Operations
KDS Full Specification

Each KDS instance configured with a prep_station. It only receives tickets for items routed to its station.

Food Safety & HACCP
Temperature log form: ingredient/equipment name, recorded temperature, target range (from config), pass/fail status, notes, timestamp, user. Required fields validated.

Daily checklists: opening/mid-service/closing. Configurable per tenant. Each checklist item has a required field (must be checked before shift can proceed) flag.

8.3 Inventory & Supply Chain — Full
Bill of Materials (BOM) Auto-Deduction
When an order_item status transitions to served (or on order close if configured), the system reads the item's recipe ingredients and creates stock_movement records with movement_type = sale_deduction for each ingredient.
BOM deduction is processed by a BullMQ job (not synchronously on the order close path) to avoid adding latency to the payment endpoint.
If an ingredient stock level drops below par_level after deduction, an inventory:low_stock event is emitted and a notification queued.
Purchase Order Workflow
States: draft → pending_approval → approved → sent → partial_received → received → cancelled
Approval workflow: if PO total exceeds tenant-configured threshold, requires approval from a user with inventory.po.approve permission before it can be sent.
On approval, system generates a PDF PO and emails it to the supplier (via BullMQ job).
Receiving: clerk scans or enters received quantities per line. System creates stock_movement records. Discrepancies flagged and stored. Invoice matching happens in finance module.
Stock Count
Admin initiates a count session for a branch. Session freezes expected quantities at count start time.
Count submitted via mobile app (barcode scan to find ingredient, enter physical count).
On count completion, system calculates variance per ingredient: variance = actual - expected. Variance report generated. Adjustment movements created for non-zero variances.

9. Security — Production Requirements
9.1 Transport & Protocol
HTTPS enforced everywhere. TLS 1.2 minimum, TLS 1.3 preferred. HTTP requests redirected to HTTPS.
WebSocket connections over WSS only.
HSTS header enforced (max-age 31536000, includeSubDomains).
Content Security Policy header on all web responses.
CORS: explicit allowlist of tenant domains. No wildcard (*) origins in production.
9.2 Authentication Security
Passwords: bcrypt with cost factor 12. Minimum 8 characters, validated server-side.
PINs: bcrypt-hashed. 4-6 digits. Cannot be the same as the user's numeric password suffix.
JWT: signed with RS256 (RSA asymmetric). Private key stored in secrets manager. Public key rotatable without downtime.
Access token lifetime: 15 minutes. Contains: sub (user_id), tenant_id, role, permissions[], branch_id, jti (unique token ID).
Refresh token: 7-day lifetime. Stored as hash in Redis with key refresh:{jti}. Rotated on every use (old token invalidated immediately). Sliding window: each refresh extends by 7 days.
Refresh token bound to user-agent + IP. Mismatch triggers revocation and admin alert.
Brute-force protection: rate limit 5 failed login attempts per IP per 15 minutes. Account lock after 10 failed attempts with admin notification.
Admin can revoke all sessions for a user: deletes all Redis keys matching refresh:{user_id}:*.
9.3 Authorization
Every API route declares required permission string (e.g., orders.create, reports.view, staff.manage).
requirePermission middleware checks JWT permissions[] claim. Returns 403 if missing.
Tenant isolation middleware: on every request, validates that the resource's tenant_id matches the JWT's tenant_id. This runs after auth, before business logic.
Super Admin routes: separate namespace /super-admin/. Only accessible with super_admin role flag in JWT. Not part of tenant JWT.
No security by obscurity: endpoints not included in JWT permissions return 403, not 404.
9.4 Data Security
All user-generated inputs sanitized server-side via fastify-sanitize-html for text fields.
All database queries use parameterized statements via Knex. No string concatenation in SQL.
File uploads: MIME type validated server-side (not just extension). Max file size enforced per type (photos: 5MB, documents: 20MB). Stored in S3 with server-side encryption (AES-256).
Card data: never stored on RMS systems. Payment tokenized by gateway before arriving at RMS API.
PII at rest: customer email, phone encrypted at application level (AES-256-GCM) with key per tenant stored in secrets manager.
GDPR/privacy: DELETE /customers/:id anonymizes personal data (replaces with anonymized_customer_{id}) and deletes name, email, phone, birthday. Order history preserved with anonymized customer_id for financial records.
9.5 Audit & Compliance
Every write operation (create, update, delete, status change) creates an audit_log entry: user_id, tenant_id, action, entity_type, entity_id, old_value_json, new_value_json, ip_address, user_agent, timestamp.
Audit log is append-only. No UPDATE or DELETE on audit_log table. Access controlled to admin and super admin only.
Audit log retention: 2 years minimum. Archivable to S3 Glacier for long-term.
All payment operations additionally logged to a payments_audit table with gateway transaction IDs.

10.2 Database Performance
All foreign keys have corresponding indexes. All WHERE clause fields on hot queries have indexes.
Critical composite indexes: (tenant_id, status) on orders; (tenant_id, is_available) on menu_items; (tenant_id, user_id, created_at) on attendance; (tenant_id, created_at) on stock_movements.
EXPLAIN ANALYZE run on every new query added to a critical path. Any query with a seq scan on a large table is rejected.
Large tables (orders, stock_movements, audit_log) partitioned by tenant_id and date range (monthly partitions). Older partitions archived and detached from main table.
Read replicas: analytics and report queries routed to a read replica. Write operations go to primary.

10.4 Scalability Design
Backend is fully stateless. All shared state lives in Redis. Any number of backend instances can serve any request.
Socket.io uses Redis adapter (@socket.io/redis-adapter) so WebSocket events emitted by one instance are received by clients connected to other instances.
BullMQ workers run as separate processes/containers from the API server. Can scale independently. Queues: notifications, pdf_generation, report_generation, email_campaigns, bom_deduction, po_email.
Horizontal scaling: add backend API instances behind Nginx load balancer. Nginx sticky sessions NOT required (stateless backend).
Vertical scaling: database scales to the PostgreSQL instance size. Horizontal read scaling via read replicas.
CDN: all static Flutter web assets served via Cloudflare CDN with far-future cache headers.
Image optimization: menu item photos resized and WebP-converted on upload via a BullMQ job. Multiple sizes stored: thumbnail (100x100), card (400x400), full (800x800).
